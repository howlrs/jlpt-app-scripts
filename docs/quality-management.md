# JLPT問題 品質管理ガイド

## 概要

JLPT問題の品質を継続的に維持・向上するための仕組み。重複・類似問題の排除、構造異常の検出、問題数の維持を自動化する。

## 品質チェック項目

### 自動検出（monitor_quality エンドポイント）

| カテゴリ | チェック項目 | 検出方法 |
|---------|------------|---------|
| **重複** | 完全一致 | sentence + 正解値の複合キー比較 |
| **重複** | 類似問題 | Levenshtein距離 ≥ 85%（カテゴリ内、文字列長フィルタ付き） |
| **構造異常** | 空括弧 | `（　　）` `（）` 等のパターンマッチ |
| **構造異常** | 選択肢数 | select_answer の要素数 ≠ 4 |
| **構造異常** | 正解キー不在 | answer の値が select_answer の key に不在 |
| **構造異常** | 空選択肢 | select_answer.value が空文字 |
| **構造異常** | 空問題文 | sub_question.sentence が空 |
| **分布** | 正解分布 | 各選択肢番号(1-4)の出現率（目標: 各25%前後） |

### 手動確認が必要な項目

| チェック項目 | 説明 |
|------------|------|
| カテゴリ別出題形式 | 漢字読みカテゴリで選択肢がひらがなか等 |
| 難易度の適切性 | N3問題にN1語彙が混入していないか |
| 日本語の自然さ | AI生成特有の不自然な表現 |
| 誤答選択肢の妥当性 | 明らかに不自然な選択肢がないか |

## 運用フロー

### 定期監視（自動）

1. Cloud Scheduler が毎週月曜 AM3:00 (JST) に `monitor_quality` を実行
2. 重複・構造異常を検出し、`execute=true` で自動削除
3. 結果を Discord Webhook に通知
4. Discord レポートを確認し、問題数が大幅に減少した場合は手動で補充

### 問題補充（手動）

1. 不足カテゴリを特定（`monitor_quality` レポートまたは `check_db` で確認）
2. `jlpt-app-scripts` のパイプラインを実行:
   ```bash
   # 問題生成（不足カテゴリに対して）
   cargo run --bin create_questions
   # パース → バリデーション → 重複排除 → 採番 → レベリング
   ./run_pipeline.sh
   # DB投入
   cargo run --bin questions_to_database
   ```
3. 投入後に `monitor_quality` でDB上の品質を最終確認

## 実績・インシデント履歴

### 2026-03-14: 空括弧問題の大量発見・削除

**問題:** N1〜N5全レベルで、sub_question.sentence に `（　　）` が含まれる不良問題を発見。漢字読みカテゴリで括弧内に漢字が入っておらず穴埋め問題化していた。文脈規定・言い換え類義・文法カテゴリにも同様の問題が広範に存在。

**影響:**
- 不良 sub_questions: 7,028件
- 不良親問題: 2,207件
- 全レベル・51カテゴリにわたって影響

**対応:**
1. 全不良問題のIDを収集
2. `bulk-delete` エンドポイントで2,207件を一括削除（10,481 → 8,108問）
3. `monitor_quality` に構造異常検出（空括弧・選択肢異常等）を追加
4. 今後の定期監視で同種の問題を自動検出・削除する仕組みを確立

**根本原因:** AI（Gemini）が問題生成時に、本来括弧内に対象語句を含めるべきところを空括弧のまま出力するケースがあった。パイプラインの `1_5_validate_questions.rs` では選択肢数・正解キー等の構造チェックのみ行っており、空括弧パターンは検出対象外だった。
